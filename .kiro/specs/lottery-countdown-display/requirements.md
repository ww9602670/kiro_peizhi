# 彩票倒计时显示与下注时机优化

## 1. 需求背景

当前系统存在以下问题：
1. 下注时机判断不准确，依赖赔率变化判断，但封盘时赔率为 null
2. SaaS 页面缺少实时的封盘倒计时和开奖倒计时显示
3. 应该直接使用平台 API 返回的 `State` 字段来判断下注状态

## 2. 功能需求

### 2.1 后端 API 改进

#### 2.1.1 获取当前期号信息
- 接口：`GET /api/v1/lottery/current-install`
- 返回数据包含：
  - `installments`: 当前期号
  - `state`: 状态码（见状态枚举表）
  - `close_countdown_sec`: 封盘倒计时（剩余秒数）
  - `open_countdown_sec`: 开奖倒计时（剩余秒数）
  - `pre_lottery_result`: 上期开奖结果
  - `pre_installments`: 上期期号
  - `template_code`: 模板代码

#### 2.1.2 状态枚举定义
平台 State 字段映射表：

| State 值 | 含义 | 是否可下注 | 前端显示 |
|---------|------|-----------|---------|
| 1 | 开盘中 | ✅ 是 | 开盘中（绿色） |
| 2 | 封盘中 | ❌ 否 | 封盘中（红色） |
| 3 | 开奖中 | ❌ 否 | 开奖中（黄色） |
| 其他 | 未知状态 | ❌ 否 | 未知（灰色） |

#### 2.1.3 下注时机优化
- Worker 使用 `State` 字段判断是否可下注
- **仅当 `State == 1` 且 `close_countdown_sec > 18` 时才允许下注**
- 下注提交前进行二次校验（条件提交策略）：
  - 校验期号一致性（内部变量 `expected_installments` 映射到 `place_bet` 的 `issue` 参数）
  - 校验状态仍为开盘（`state == 1`）
  - 校验剩余时间充足（`close_countdown_sec > 18`）
  - 提交失败时记录错误码，不进行重试（遵循"Confirmbet 零重试"原则）
- 移除基于赔率变化的判断逻辑

### 2.2 前端显示改进

#### 2.2.1 仪表盘实时倒计时
在仪表盘页面添加实时倒计时显示：
- 当前期号
- 封盘倒计时（剩余秒数）
- 开奖倒计时（剩余秒数）
- 当前状态（根据 State 字段显示：开盘中/封盘中/开奖中/未知）

#### 2.2.2 策略页面倒计时
在策略页面也显示倒计时信息，方便用户监控

#### 2.2.3 自动刷新与降级策略
- 每秒更新倒计时显示（客户端计算）
- 每 5 秒调用 API 获取最新数据
- API 异常时显示"数据延迟"提示和最后更新时间
- 使用共享 hook 避免多页面重复轮询

## 3. 验收标准

### 3.1 后端
- [ ] 新增 `/api/v1/lottery/current-install` 接口
- [ ] Worker 使用 `State` 判断下注时机（State == 1 且 close_countdown_sec > 18）
- [ ] 下注提交前进行二次校验（期号一致性 + 状态 + 时间）
- [ ] 下注请求使用 `place_bet(issue=expected_installments, betdata=...)` 绑定期号
- [ ] 遵循"Confirmbet 零重试"原则（不实现重试逻辑）
- [ ] 移除基于赔率的判断逻辑
- [ ] **下注成功率提升（量化指标）**：
  - 统计窗口：连续 3 天（工作日，排除节假日）
  - 样本量：每天至少 50 次下注机会（定义：State=1 且 close_countdown_sec>18 的期数）
  - 基线：当前"赔率已改变"错误率（采样时间：部署前 3 天）
  - 目标：错误率下降 >= 80%
  - 异常剔除：网络超时、平台维护期间的数据不计入

### 3.2 前端
- [ ] 仪表盘显示实时倒计时
- [ ] 策略页面显示实时倒计时
- [ ] 倒计时每秒更新
- [ ] 状态文字清晰（开盘中/封盘中/开奖中/未知）
- [ ] 状态颜色正确（开盘=绿色，封盘=红色，开奖=黄色，未知=灰色）
- [ ] API 异常时显示降级提示

### 3.3 集成测试
- [ ] Chrome MCP 验证倒计时显示正常
- [ ] 验证下注只在 State=1 且 close_countdown_sec>18 时执行
- [ ] 验证封盘时不会尝试下注
- [ ] 验证下注成功率达到目标（错误率下降 >= 80%）
- [ ] 验证 API 响应符合 Envelope 格式

## 4. 技术约束

- 使用现有的 JNDAdapter 调用平台 API
- 前端使用 React hooks 实现倒计时
- 遵循契约先行规范（先定义 schema）
- 遵循 Proxy-Only 开发模式

## 5. 优先级

P0 - 高优先级（影响下注成功率）
