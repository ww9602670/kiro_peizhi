# 加拿大PC28自动投注托管SaaS平台 — 需求文档

## 概述
构建一个 SaaS 平台，作为第三方博彩网站的自动投注代理工具。平台本身不是博彩网站，而是帮助操作者自动登录博彩网站、按预设策略自动下注、记录投注与结算数据的托管工具。

## 一期/二期边界

### 一期范围
- 单平台接入（目标平台 + 两种盘口 JND28WEB/JND282）
- 平注策略 + 自定义马丁策略
- 基础对账与结算
- 操作者/管理员仪表盘
- 风控阀门（限额、止损止盈、登录风控）
- 告警通知（登录失败、下注失败、对账异常等）
- 策略注入接口定义（仅接口规范 + 文档，不实现 AI 策略）

### 二期范围（本文档仅做架构预留，不实现）
- AI 策略实现
- 多平台扩展（其他 PC28 平台接入）
- 高级风控（基于历史数据的智能风控）
- 操作审计日志的高级查询与导出

## 1. 用户系统

### 1.1 管理员账户管理
**用户故事**: 作为管理员，我需要创建和管理操作者账户，以便控制平台的用户访问。

**验收标准**:
- 管理员可以创建操作者账户（用户名、密码、最大可绑定博彩账号数、到期日期）
- 管理员可以查看所有操作者列表及其状态（正常/过期/禁用）
- 管理员可以修改操作者的最大可绑定账号数和到期日期
- 管理员可以禁用/启用操作者账户
- 到期账户自动标记为过期状态
- 管理员操作记录审计日志（创建/修改/禁用操作者的时间和操作人）

### 1.2 操作者登录与会话管理
**用户故事**: 作为操作者，我需要登录平台以使用自动投注功能。

**验收标准**:
- 操作者使用用户名+密码登录
- 登录后颁发 JWT Token，有效期 24 小时
- Token 过期前 30 分钟自动续期（静默刷新）
- 同一操作者同时只允许一个活跃会话（后登录踢掉前会话）
- 过期账户无法登录，提示联系管理员
- 禁用账户无法登录
- 登录失败记录日志（IP、时间、失败原因）

### 1.3 博彩账号绑定
**用户故事**: 作为操作者，我需要绑定博彩网站账号，以便系统代我自动投注。

**验收标准**:
- 操作者可以绑定博彩网站账号（账号+密码）
- 绑定时必须选择盘口类型：JND网盘（JND28WEB）或 JND2.0（JND282）
- 盘口类型决定 lotteryType 参数和结算规则
- 默认每个操作者只能绑定1个博彩账号
- 管理员可以调整操作者的最大绑定数量
- 绑定时系统尝试登录验证账号有效性（最多重试2次）
- 登录需自动识别并填写图片验证码
- 博彩账号密码在前端展示时脱敏（仅显示前2位+****）
- 博彩账号密码明文存储（用户明确不要求加密）

### 1.4 数据隔离
**验收标准**:
- 操作者之间的数据完全隔离（投注记录、策略、博彩账号、盈亏数据）
- 操作者只能查看和操作自己的数据
- 管理员可以查看所有操作者的数据（只读）
- 所有 API 请求通过 JWT 中的操作者 ID 进行数据过滤

## 2. 博彩平台集成

### 2.1 平台登录
**用户故事**: 作为系统，我需要自动登录博彩网站，以便执行后续的投注操作。

**验收标准**:
- 系统通过博彩网站表单接口进行登录（字段：txtName/txtPwd/txtVerify）
- 图片验证码自动识别（仅登录时出现）
- 登录成功后获取 Session Cookie（含 token 字段），后续所有请求携带此 Cookie
- 会话通过心跳接口（/Member/Online）保持活跃，固定每 75 秒调用一次
- 心跳连续失败 3 次判定会话失效，触发重新登录
- 会话过期时通过 Token 刷新接口尝试恢复，失败则重新登录
- 登录失败时记录错误并通知操作者

### 2.2 登录风控
**验收标准**:
- 登录失败（含验证码识别失败）后等待递增间隔重试：第1次等30秒，第2次等60秒，第3次等120秒
- 连续登录失败 3 次后暂停该账号的自动登录，等待 10 分钟后重试
- 连续登录失败 5 次后停止该账号的自动登录，标记为"登录异常"，通知操作者手动处理
- 验证码识别失败单独计数，连续失败 5 次通知操作者"验证码识别异常"
- 所有登录尝试记录日志（时间、结果、失败原因、重试次数）

### 2.3 开奖数据获取
**用户故事**: 作为系统，我需要实时获取开奖数据，以便进行结算和策略判断。

**验收标准**:
- 通过 GetCurrentInstall 接口获取当前期号、倒计时、上期开奖结果
- 通过 Lotteryresult 接口批量获取历史开奖数据（用于策略预加载和补结算）
- 轮询策略：每 5 秒轮询一次 GetCurrentInstall，检测到新期号时触发结算+策略计算
- 获取数据包含：期号（Installments）、开奖号码（OpenResult，格式"球1,球2,球3"）、开奖时间
- 已知停盘时间（19:56-20:33, 06:00-07:00）自动暂停轮询，改为每 30 秒探测一次
- 随机停盘检测：连续 3 次轮询返回 State≠1 且期号未变化，判定为随机停盘
- **注意**：随机停盘的检测信号需在开发阶段通过 Chrome 抓包实际验证，确认后再固化检测逻辑

### 2.4 封盘状态检测
**用户故事**: 作为系统，我需要在下注前检测封盘状态，避免在封盘后下注失败。

**验收标准**:
- 通过 GetCurrentInstall 接口的 State 字段判断：State=1 为开盘可投注，其他值为封盘
- CloseTimeStamp 字段表示距封盘秒数，用于计算下注时机
- 每次下注前检查当期封盘状态
- 封盘后不再尝试下注，等待下一期

### 2.5 API 调用频率控制
**验收标准**:
- 每个博彩账号的 API 调用频率限制：
  - GetCurrentInstall：最多每 5 秒 1 次
  - Loaddata（赔率）：最多每 3 秒 1 次
  - Confirmbet（下注）：每期最多调用 1 次（合并多策略的下注到一次请求）
  - QueryResult（余额）：最多每 10 秒 1 次
  - Topbetlist（注单）：最多每 10 秒 1 次
  - Online（心跳）：固定每 75 秒 1 次
- 超出频率限制的请求排队等待，不丢弃
- API 调用超时设置：普通接口 10 秒，下注接口 15 秒
- 超时后重试最多 2 次，间隔 3 秒

## 3. 投注玩法

### 3.1 支持的玩法类型
**用户故事**: 作为操作者，我需要选择不同的玩法进行投注。

**验收标准**:
- 大小（DX）：和值 ≥14 为大（DX1），≤13 为小（DX2）
- 单双（DS）：和值为奇数为单（DS3），偶数为双（DS4）
- 特码/和值（HZ）：猜具体和值 0-27（HZ1~HZ28，注意 HZ1 对应和值 0）
- 组合（ZH）：大单（ZH7）、大双（ZH8）、小单（ZH9）、小双（ZH10）
- 极值（JDX）：极大 22-27（JDX5）、极小 0-5（JDX6）
- 色波（SB）：红波（SB1）、绿波（SB2）、蓝波（SB3）
- 豹子（BZ）：三球相同（BZ4）
- 单球号码（B1QH/B2QH/B3QH）：第一/二/三球猜具体数字 0-9
- 单球两面（B1LM/B2LM/B3LM）：第一/二/三球的大/小/单/双
- 龙虎和（LHH）：龙（LHH_L）、虎（LHH_H）、和（LHH_HE）
- 所有玩法 KeyCode 与博彩平台 API 一致（详见 platform-api.md 玩法代码映射表）

### 3.2 特殊结算规则
**验收标准**:

**盘口类型区分：** 平台存在两种盘口，结算规则不同，操作者绑定博彩账号时必须选择盘口类型。

#### JND28WEB（JND网盘）
- 和值 13/14 时，所有玩法（包括大小、单双、组合）均按显示赔率正常结算
- 无退款规则，所有玩法统一按赔率结算

#### JND282（JND2.0盘）
- 和值 14 时：
  - 下注"大"（DX1）或"双"（DS4）→ 退款（返还本金，视为和局）
  - 下注"大双"（ZH8）→ 退款（返还本金，视为和局）
  - 其他玩法正常结算
- 和值 13 时：
  - 下注"小"（DX2）或"单"（DS3）→ 退款（返还本金，视为和局）
  - 下注"小单"（ZH9）→ 退款（返还本金，视为和局）
  - 其他玩法正常结算
- 退款期对马丁策略的影响：不算命中也不算未命中，马丁序列位置不变

**系统要求：**
- 操作者绑定博彩账号时必须选择盘口类型（网盘/2.0）
- 结算引擎根据盘口类型应用不同的结算规则
- 盘口类型决定 API 请求中的 lotteryType 参数（JND28WEB 或 JND282）

## 4. 投注策略

### 4.1 平注策略
**用户故事**: 作为操作者，我需要设置平注策略，每期固定买固定号码和金额。

**验收标准**:
- 操作者选择玩法和具体投注内容（如：买大 DX1）
- 设置固定投注金额（最小 1 元，最大由风控阀门限制）
- 每期按相同配置自动下注
- 策略可随时启动/暂停/停止
- 策略状态：未启动 → 运行中 → 暂停 → 已停止

### 4.2 自定义马丁策略
**用户故事**: 作为操作者，我需要设置自定义马丁格尔策略，按自定义倍率序列追注。

**验收标准**:
- 操作者自定义倍率序列（如：[1, 2, 4, 8, 16]，表示基础金额的倍数）
- 操作者设置基础金额（如：10 元，则序列实际金额为 [10, 20, 40, 80, 160]）
- 策略可命名并保存，与操作者账号绑定
- 命中 → 重置到序列第1级，继续下一期
- 完成整个序列仍未命中 → 重置到序列第1级，继续下一期（不自动停止，除非操作者手动停止）
- JND2.0 盘口退款期 → 马丁序列位置不变，下一期继续当前级别
- 停盘期间暂停 → 恢复后从当前序列位置继续
- 策略可随时启动/暂停/停止
- 支持多个已保存的马丁策略模板
- 马丁序列跑完一轮重置时记录日志（含当前轮次亏损总额）

### 4.3 策略注入接口（一期仅定义接口规范）
**用户故事**: 作为开发者，我需要一个文档完善的策略注入接口，以便未来接入AI策略或自定义扩展策略。

**验收标准**:
- 定义清晰的策略接口规范：
  - 输入：当前期号、历史开奖数据（最近 N 期）、账户余额、当前策略状态（马丁级别等）
  - 输出：投注指令列表（每条包含：KeyCode、金额）或"本期不投注"
- 提供策略注入的详细开发文档（Markdown 格式，含示例代码）
- 策略接口支持同步调用
- 新策略可通过 Python 模块注册，无需修改核心投注引擎代码
- 一期不实现 AI 策略，仅预留接口

### 4.4 多策略并行
**用户故事**: 作为操作者，我需要同时运行多个策略，比如一个买大小，另一个买特码。

**验收标准**:
- 同一博彩账号可同时运行多个策略
- 不同策略可选择不同玩法，也可选择相同玩法
- 各策略独立运行，互不干扰
- 每个策略产生下注信号时生成唯一幂等 ID（格式：{期号}-{策略ID}-{KeyCode}）
- 防重复下注基于幂等 ID：同一幂等 ID 只允许下注一次
- 不同策略即使产生相同 KeyCode 的下注信号，也视为不同的下注需求，均需执行
- 同一期多个策略的下注请求合并为一次 API 调用（减少请求频率）
- 操作者可查看每个策略的独立运行状态和盈亏

## 5. 投注执行引擎

### 5.1 投注时机控制
**用户故事**: 作为操作者，我需要控制每期的下注时机。

**验收标准**:
- 默认在开盘后 30 秒下注
- 最早可设置为开盘后 5 秒
- 最晚不超过封盘前 10 秒（基于 CloseTimeStamp 计算）
- 操作者可自定义下注时间点（秒数）
- 下注时间点在策略配置中设置，每个策略可独立配置

### 5.2 投注前置检查
**用户故事**: 作为系统，我需要在下注前进行多重检查，确保投注安全。

**验收标准**:
- 检查流程（按顺序）：
  1. 策略信号：策略是否产生了本期的下注指令
  2. 门控检查：会话是否有效、策略是否处于运行状态、操作者账户是否正常
  3. 风控阀门检查：余额是否充足、是否超过单笔/单日限额、是否触发止损/止盈
  4. 封盘状态检查：State=1 且 CloseTimeStamp > 10（确保在封盘前有足够时间完成下注）
  5. 赔率获取：获取最新赔率（赔率为 0 表示该玩法暂停，跳过）
  6. 幂等检查：该幂等 ID 是否已下注
- 任一检查失败则跳过本期投注并记录原因（含具体失败步骤）
- **下注失败的主要原因是封盘导致赔率为 0**，因此封盘检测是最关键的前置检查
- 下注时必须提交当前最新赔率
- 下注失败（succeed≠1）时根据错误码决定处理方式：
  - succeed=5（赔率变动/封盘）：大概率是封盘导致，记录日志，不重试（本期已过）
  - succeed=10（参数错误）：记录错误，不重试
  - succeed=18（限额为0/玩法暂停）：记录错误，不重试
  - 其他：记录错误，不重试

### 5.4 下注合并
**验收标准**:
- 同一期、同一博彩账号的多个策略产生的下注指令合并为一次 Confirmbet API 调用
- 合并方式：将多个策略的 betdata 放入同一个请求（betdata[0]、betdata[1]...）
- 每条 betdata 包含：Amount（金额）、KeyCode（玩法代码）、Odds（当前赔率）
- 不同策略产生的相同 KeyCode 下注不合并金额，作为独立的 betdata 条目
- 合并请求失败时整体失败，不拆单重试（因为失败主要原因是封盘，拆单也无意义）
- 每条 betdata 的结果通过幂等 ID 独立追踪

### 5.3 停盘处理
**用户故事**: 作为系统，我需要在停盘期间自动暂停并在恢复后继续。

**验收标准**:
- 已知停盘时间（19:56-20:33, 06:00-07:00）：提前 1 分钟进入停盘等待模式
- 随机停盘检测：连续 3 次轮询（间隔 5 秒）返回 State≠1 且期号未变化，判定为随机停盘
- 停盘期间：暂停所有投注，改为每 30 秒探测一次 GetCurrentInstall
- 恢复检测：探测到 State=1 且出现新期号，判定为恢复
- 恢复后自动继续投注
- 马丁策略在停盘恢复后从当前序列位置继续（不重置）
- 停盘/恢复事件记录在日志中（时间、类型、持续时长）

## 6. 风控阀门

### 6.1 投注限额
**用户故事**: 作为操作者，我需要设置投注限额，控制资金风险。

**验收标准**:
- 限额维度：按博彩账号配置（每个博彩账号独立的限额设置）
- 博彩平台自身有玩法限额（不同玩法的单注最大金额不同），系统需从平台获取或配置
- 单笔限额：每次下注的最大金额（默认跟随平台限额，操作者可设置更低值）
- 单期限额：每期所有策略合计下注的最大金额（默认无限制）
- 单日限额：每日（自然日）累计下注的最大金额（默认无限制）
- 超过博彩平台限额时：提示操作者"该玩法限额为 X 元，是否切换盘口或新增账号"
- 超过自设限额时：跳过本期投注，记录日志并通知操作者

### 6.2 止损止盈
**验收标准**:
- 止损止盈维度：按策略配置（每个策略独立的止损止盈设置）
- 计算口径：策略的净盈亏（实际盈利 - 实际亏损，退款不计入）
- 止损线：策略当日净亏损达到设定值时，自动暂停该策略
- 止盈线：策略当日净盈利达到设定值时，自动暂停该策略
- 触发止损/止盈时通知操作者（含策略名称和触发值）
- 操作者可手动解除止损/止盈状态，恢复该策略的投注
- 止损/止盈默认关闭，操作者可自行开启并设置阈值
- 每日 0:00 自动重置当日盈亏统计

### 6.3 余额不足处理
**验收标准**:
- 下注前检查博彩账号余额是否 ≥ 本期计划下注金额
- 余额不足时跳过本期投注，记录日志
- 连续 3 期余额不足时暂停该博彩账号的所有策略，通知操作者"余额不足"

### 6.4 全局熔断（Kill Switch）
**验收标准**:
- 操作者可一键停止某个博彩账号下的所有策略（紧急停止）
- 管理员可一键停止所有操作者的所有策略（全局紧急停止）
- 熔断触发后所有待执行的下注指令立即取消
- 熔断状态需手动解除才能恢复投注
- 熔断事件记录日志（触发人、时间、影响范围）

### 6.5 模拟模式（影子模式）
**验收标准**:
- 策略支持"模拟模式"运行：执行所有策略逻辑但不实际调用 Confirmbet 下注
- 模拟模式下记录"虚拟注单"（标记为模拟），计算虚拟盈亏
- 操作者可查看模拟模式的投注记录和盈亏统计
- 用于验证策略逻辑正确性后再切换为实盘模式
- 模拟/实盘模式切换在策略配置中设置

## 7. 结算与记录

### 7.0 注单状态机
**验收标准**:
- 注单状态流转：
  ```
  待投注 → 投注中 → 投注成功 → 结算中 → 已结算
                  ↘ 投注失败
                                          ↘ 对账异常
  ```
- 状态定义：
  - 待投注：策略产生下注信号，等待执行
  - 投注中：已发送 Confirmbet 请求，等待响应
  - 投注成功：Confirmbet 返回 succeed=1
  - 投注失败：Confirmbet 返回 succeed≠1 或请求超时
  - 结算中：开奖后正在计算盈亏
  - 已结算：开奖后计算盈亏完成
  - 对账异常：已结算但余额校验不通过
- 状态变更时记录日志（时间、旧状态、新状态、触发原因）
- 幂等 ID 作为注单唯一标识，数据库唯一约束

### 7.1 结算处理
**用户故事**: 作为系统，我需要自动获取结算结果并计算盈亏。

**验收标准**:
- 每期开奖后（检测到新期号时）触发结算流程
- 根据开奖结果和赔率计算每笔投注的理论盈亏
- 结算规则根据盘口类型（JND28WEB/JND282）应用不同的 13/14 规则（详见 3.2）
- 通过 QueryResult 接口获取账户余额（accountLimit）、今日盈亏（Result）
- 余额交叉校验：理论余额与实际余额差额 ±1 元以内视为正确（精度为元，因平台可能有小数截断）
- 校验不通过时将注单标记为"对账异常"，记录差额，通知操作者
- 对账异常不自动停止策略，但在仪表盘醒目提示
- 结算数据持久化到数据库

### 7.2 投注记录展示
**用户故事**: 作为操作者，我需要查看详细的投注记录。

**验收标准**:
- 投注记录表格包含：期号、下注时间、下注号码（KeyCode + 中文名）、金额、开奖号码、盈亏结果
- 按期号倒序排列
- 异常记录（对账异常、下注失败等）高亮显示
- 支持按日期范围筛选
- 支持按策略筛选
- 分页展示，每页 50 条

### 7.3 操作者仪表盘
**用户故事**: 作为操作者，我需要一个仪表盘查看整体投注状况。

**验收标准**:
- 显示当前余额（每 30 秒从博彩平台刷新）
- 显示当日盈亏
- 显示总盈亏
- 显示当前运行中的策略列表及其状态（运行中/暂停/已停止/异常）
- 显示每个策略的独立盈亏
- 显示最近 20 条投注记录
- 显示告警信息（登录异常、对账异常、余额不足等）

### 7.4 管理员仪表盘
**用户故事**: 作为管理员，我需要查看所有操作者的运营数据。

**验收标准**:
- 查看所有操作者列表及其状态、总盈亏、当日盈亏
- 可下钻查看某个操作者的策略详情
- 可查看某个策略的独立盈亏
- 可查看某个策略的详细投注记录
- 操作者账户管理（创建/修改/禁用）

## 8. 告警通知

### 8.1 通知渠道
- 一期仅支持站内通知（仪表盘实时显示 + 页面内弹窗/横幅提示）
- 不支持外部渠道（邮件、短信、Telegram 等为二期范围）

### 8.2 操作者告警
**验收标准**:
- 以下事件触发操作者告警：
  - 博彩账号登录失败（含失败原因和重试状态）
  - 验证码识别连续失败（含失败次数）
  - 博彩账号会话断线（含自动重连状态）
  - 下注失败（含失败原因、期号、错误码）
  - 对账异常（含差额和期号）
  - 余额不足（含当前余额和所需金额）
  - 止损/止盈触发（含策略名称和触发值）
  - 马丁序列跑完一轮重置（含策略名称和本轮亏损总额）
  - 博彩平台限额触发（含玩法和限额值，提示切换盘口或新增账号）
- 告警信息包含：时间、类型、详细描述
- 告警列表支持标记已读
- 未读告警在仪表盘顶部醒目显示数量

### 8.3 系统级告警（管理员）
**验收标准**:
- 管理员仪表盘显示系统级告警：
  - 博彩平台 API 大面积不可用（超过 30% 的活跃账号请求失败）
  - 单个操作者连续异常（连续 5 期下注失败）
- 告警信息在管理员仪表盘醒目展示

## 9. 多平台扩展（架构预留，一期不实现多平台）

### 9.1 平台抽象层
**用户故事**: 作为开发者，我需要一个平台抽象层，以便未来接入其他PC28平台。

**验收标准**:
- 博彩平台交互逻辑抽象为 Python 抽象基类（ABC），包含以下方法：
  - `login(account, password)` → Session
  - `get_current_install()` → 期号/倒计时/上期结果
  - `load_odds(issue)` → 赔率字典
  - `place_bet(issue, bets)` → 下注结果
  - `query_balance()` → 余额/盈亏
  - `get_bet_history(count)` → 投注记录
  - `get_lottery_results(start, rows)` → 开奖结果
  - `heartbeat()` → 心跳状态
- 第一版实现目标平台适配器（含两种盘口 JND28WEB/JND282）
- 平台配置（base_url、lotteryType、停盘时间、玩法代码映射）通过配置文件管理
- 新平台接入只需继承抽象基类 + 提供配置文件

## 10. 非功能需求

### 10.1 并发性能
- 支持 1000 个博彩账号同时在线自动投注
- 每个博彩账号独立运行一个投注任务（异步协程）
- 单个账号异常不影响其他账号（进程/协程隔离）
- 投注引擎使用 Python asyncio 实现并发调度

### 10.2 可靠性
- 投注引擎异常自动恢复（单账号级别重启）
- 网络异常时按 2.5 节的频率控制策略重试
- 关键操作日志：下注请求/响应、结算结果、余额变动、策略状态变更
- 数据库事务保证投注记录和结算数据的一致性

### 10.3 可观测性
- 所有投注操作记录结构化日志（JSON 格式）
- 日志字段：时间戳、操作者ID、博彩账号、期号、操作类型、结果、耗时
- 异常事件实时记录并触发告警
- 系统指标：活跃账号数、每分钟下注次数、API 成功率、平均响应时间
